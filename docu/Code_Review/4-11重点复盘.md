# Code Review 4-11（Backend）

## 1. 目前已实现的功能

### 1.1 用户与认证模块
- 用户注册、登录（JWT 鉴权）。
- 获取个人信息、修改个人信息。
- 上传用户头像（含文件大小、扩展名、魔数校验）。
- 获取“我发布的文章”分页列表。
- 管理员仪表盘统计接口、管理员上传站点图片接口。

### 1.2 文章模块
- 文章发布、编辑、删除、按 ID 获取详情。
- 全部文章列表、搜索文章、置顶/推荐切换与列表获取。
- 点赞/取消点赞（含事务处理）。
- 阅读量增加接口。
- 文章封面上传（复用图片上传服务）。

### 1.3 评论与分类标签模块
- 评论发布、删除、按文章获取评论列表。
- 分类 CRUD（管理员写操作受限）。
- 标签 CRUD（当前接口可用）。

### 1.4 基础设施
- EF Core Code First + 迁移文件。
- 关系建模：`ArticleTag`、`ArticleLike` 复合主键。
- 用户名/邮箱唯一索引。
- Swagger 文档、静态文件托管、CORS、JWT 中间件链路。

## 2. 后续拓展方向（建议优先级）

### 高优先级（建议近期完成）
- 增加全局异常处理中间件，统一错误码与返回结构。
- 将文章搜索改为数据库侧过滤（避免全表加载到内存）。

### 低优先级

- 增加 OpenTelemetry 链路追踪和请求 ID。

## 3. Review 发现的问题

### [P1] 标签管理缺少管理员权限保护
- 位置：`backend/CuteBlogSystem/Controller/TagsController.cs`
- 现象：`POST/PUT/DELETE` 未加 `[Authorize(Roles = "Admin")]`。
- 风险：普通用户可直接创建/修改/删除标签，越权。

### [P1] 更新标签接口路由参数未被使用
- 位置：`backend/CuteBlogSystem/Controller/TagsController.cs`
- 现象：路由定义 `PUT {tagId}`，方法仅接收 `updatedTag`，未使用 `tagId`。
- 风险：请求语义和实际行为不一致，容易被错误调用或造成数据误改。

### [P1] 管理员上传图片接口缺少 `TryParse` 失败分支
- 位置：`backend/CuteBlogSystem/Controller/AuthController.cs` -> `UploadAdminImage`
- 现象：调用了 `TryParse` 但未根据 `success` 做拦截，解析失败时默认 `userIdInt=0` 继续执行业务。
- 风险：错误用户上下文被带入服务层，产生误报或潜在安全问题。

### [P2] 搜索逻辑在服务层做内存过滤
- 位置：`backend/CuteBlogSystem/Service/ArticleService.cs` -> `SearchArticlesAsync`
- 现象：先 `GetArticlesAsync()` 全量取出，再在内存中 `Where`。
- 风险：数据量增大后性能明显下降，内存占用上升，分页失真。

### [P2] CORS 仅放行本地开发地址
- 位置：`backend/CuteBlogSystem/Program.cs`
- 现象：仅允许 `localhost:5173 / 127.0.0.1:5173`。
- 风险：上线后前端域名/隧道地址无法访问 API（环境切换成本高）。

### [P2] 缺少全局异常处理与统一错误映射
- 位置：整体架构
- 现象：主要依赖 Controller/Service 层局部 try-catch。
- 风险：异常返回格式不稳定，调试和前端处理成本高。

### [P3] 代码注释与部分字符串存在编码异常（乱码）
- 位置：多处（如 `ArticleService.cs`、`Program.cs`）
- 风险：可维护性下降，团队协作理解成本高，文档化困难。

## 4. 修改方式（落地思路）

### 4.1 权限与接口边界
1. 在 `TagsController` 的 `Create/Update/Delete` 增加 `[Authorize(Roles = "Admin")]`。  
2. `UpdateTag` 显式接收 `tagId`，并与请求体 ID 做一致性校验（或直接只信任路由 ID）。

### 4.2 认证与输入校验
1. `UploadAdminImage` 参考其他接口写法：`TryParse` 失败直接 `Unauthorized` 返回。  
2. 对上传接口统一 `ResponseCode -> HttpStatus` 映射策略，避免分散 if/else。

### 4.3 搜索性能优化
1. 在 Repository 层新增可组合查询（关键词、分类、标签、分页）。  
2. 让 EF 生成 SQL 执行过滤、排序、分页，Service 仅做参数校验和组装 DTO。

### 4.4 稳定性与可观测性
1. 增加全局异常中间件（统一记录日志 + 统一 `ApiResponse`）。  
2. 对关键链路打结构化日志：`UserId`、`ArticleId`、`TraceId`。  
3. 增加基础限流/防刷策略（特别是阅读量、点赞、评论接口）。

### 4.5 工程规范
1. 统一文件编码为 UTF-8（无 BOM/带 BOM 按团队规范一致即可）。  
2. 将“开发环境 CORS”与“生产环境 CORS”拆分到配置文件。  
3. 增加最小集成测试：认证、发布文章、评论、点赞、权限拦截。

---

## 5. 建议的下一步执行顺序
1. 先修权限问题（标签写接口 + `UploadAdminImage` 解析分支）。  
2. 再做搜索下沉到数据库（收益最大）。  
3. 最后补全全局异常与日志规范，顺带清理乱码编码。

这样能以最小改动先把“安全与正确性”兜住，再提升性能和可维护性。
