1. 2026-3-26 实体类迁移数据库失败问题：

   ①问题1：迁移时连接字符串失败。发现是更改连接字符串后没有进行保存，保存后解决连接失败问题，但再次出现报错。

   ②问题2：迁移时出现联级删除导致无法迁移。最后通过配置MyDbContext，将Comment -> User，Comment -> Article，Commetn -> ParentComment，Article -> User，Article -> Category设置为Restrict后成功完成迁移。

   配置代码示例如下：

   ```c#
   // Comment -> User
   modelBuilder.Entity<Comment>()
       .HasOne(c => c.User)
       .WithMany(u => u.Comments)
       .HasForeignKey(c => c.UserId)
       .OnDelete(DeleteBehavior.Restrict); 
   ```

   2026-3-26 Dto设计问题：

   ​	在写完控制器后，着手开发第一个功能时，我开始想着怎么去设计DTO，比如用户修改自己的签名信息，是单独做成一条dto还是整条用户信息作为一个大的dto一并发送。

   解决：我想明白了，DTO实质上是面向功能开发的实体类截取字段组成的，这个功能需要多少字段，我的DTO就应该设计成从实体类里面截取多少字段。也就是说，修改信息是一个单独的功能，那我设计的DTO就应该同时包含可修改的信息，比如签名bio，昵称NickName等。

   2026-3-27 Token设计问题：

   ​	我发现如果只使用JWT来维持对话状态，会导致在JWT有效期有限的情况下，用户需要重复登录才能获取新的Token，这样的设计十分不合理。

   解决：新增一张RefreshTokens表，在用户登录后，返回一个RefreshToken(随机64位数，有效期比JWT长很多倍)然后存到数据库里，用户的JWT过期之后，服务器判断用户的RefreshToken是否过期，如果没过期，就签发新的JWT。

   2026-3-27 对于用户登录失败回复信息设计问题：

   ​	我把输错密码和用户不存在两种分开检测，作为两种不同信息回复：

   ```c#
   if (user == null)
   {
       return new ApiResponse(false, "登录失败！用户名不存在！");
   }
   
   // 验证密码
   bool isPasswordValid = BCrypt.Net.BCrypt.Verify(loginDTO.Password, user.PasswordHash);
   if (!isPasswordValid)
   {
       return new ApiResponse(false, "登录失败！密码错误！");
   }
   ```

   ​	GPT跟我讲可能会带来枚举攻击的风险，我也突然想起来好像市面上的登录功能都是回复“用户名或密码错误”，没有见过是单独回复哪一条错误的。

   解决：不做大的改动，将两条ApiResponse的信息全改为“登录失败！用户名或密码错误！”

   2026-3-28 封面上传功能设计问题：
   
   ​	在封面上传前文章并没有上传，这导致数据库里没有这条文章的ArticleId，导致封面url无法绑定到文章id上。
   
   解决：删除上传封面功能里对于数据库操作，仅进行文件保存并返回url，在前端进行文章发布时再将封面url传回后端进行与文章绑定添加到数据库中。
